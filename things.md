🏗 Архитектура мессенджера TellThink

Версия: 1.1
Статус: В разработке
Дата: 19.06.2026

1. Аннотация проекта

TellThink — это кроссплатформенный клиент-серверный мессенджер. 
Проект фокусируется на базовой функциональности обмена сообщениями в реальном времени, поддержке чатов и удобной работе с вложениями (медиа и документы).

2. Технологический стек

2.1 Backend (Core)
- Язык: Go (Golang). Высокая производительность и нативная поддержка многопоточности.
- Протокол API: REST API (фреймворк Gin) и WebSockets (Gorilla) для доставки сообщений в реальном времени.
- База данных: PostgreSQL (хранение пользователей, истории сообщений, метаданных чатов).
- Кэш/Session Store: Redis (быстрый доступ, pub/sub для доставки сообщений).
- Файловое хранилище: Cloud.ru Evolution Object Storage (S3-compatible).

> [!IMPORTANT]
> **Особенности работы с медиа (Pre-Signed URLs):** 
> Cloud.ru Evolution строго требует `tenant_id` даже для публичных бакетов, что блокирует прямые анонимные загрузки из Flutter. 
> Для решения этой проблемы бэкенд использует `s3.PresignClient` и динамически генерирует **временные подписанные ссылки** (Pre-Signed URLs, валидные 4 часа) при каждом запросе сообщений. Это также повышает безопасность, исключая постоянный публичный доступ к файлам.

2.2 Client Side
- Фреймворк: Flutter (кроссплатформенная разработка мобильных приложений).
- Сетевой слой: REST (Dio) для обычных запросов и WebSockets для реалтайм-обновлений.
- Вложения файлов: Реализовано через пакеты `file_picker` (для выбора любых форматов файлов) и `url_launcher`. Скачивание файлов делегируется встроенному системному браузеру, что решает проблемы с разрешениями (Permissions) на мобильных устройствах и позволяет скачивать файлы в фоне.
- Локальная БД: SQLite (для кэширования истории переписки).

3. DevOps и Инфраструктура

3.1 Автоматизация (IaC)
- Docker Compose: Оркестрация базовых сервисов (Бэкенд + БД + Redis) в рамках одного сервера.

3.2 CI/CD Pipeline (GitHub Actions)
- Build: Автоматическая сборка бинарного файла сервера и проверка кода.
- Deploy: Автоматическое обновление приложения на сервере при пуше в ветку main.

4. План разработки (Roadmap)

- Этап 1 (MVP): Регистрация, авторизация, чат 1-на-1, история сообщений, отправка текстовых сообщений. (Выполнено)
- Этап 2: Отправка и скачивание любых файлов через S3, настройка автоматического CI/CD, отображение статусов онлайна и печати в реальном времени. (Выполнено)
- Этап 3 (Текущий): Групповые чаты (создание, управление ролями, удаление, выход из группы), редактирование и удаление сообщений. (Частично выполнено)
- Этап 4: Система уведомлений (Push), расширенные профили пользователей, мультипрофильность на сервере.
